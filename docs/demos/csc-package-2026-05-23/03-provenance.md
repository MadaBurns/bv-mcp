## Re-derivation — public-tooling commands

Every finding above can be re-derived without BlackVeil involvement, from any machine with `curl`, `dig`, and `jq`. The commands below produce the exact evidence the verification matrix references.

### Subdomain enumeration (CT logs)

```bash
curl -A "Mozilla/5.0" "https://crt.sh/?q=%25.openai.com&output=json" \
  | jq -r '.[].name_value' \
  | tr ',' '\n' \
  | grep '\.openai\.com$' \
  | sort -u
```

Substitute `openai.com` for any of `anthropic.com`, `x.ai`, `cohere.com`, etc.

### DNS-layer dangling-CNAME proof (any of the 12 DNS-layer findings)

```bash
# Step 1: confirm the CNAME chain
dig +short @one.one.one.one CNAME slack.openai.com
# → chatgpt-slack.jollybeach-2c1b6a13.centralus.azurecontainerapps.io.

# Step 2: confirm the target is NXDOMAIN
dig +noall +comments @one.one.one.one A chatgpt-slack.jollybeach-2c1b6a13.centralus.azurecontainerapps.io \
  | grep status:
# → ;; ->>HEADER<<- opcode: QUERY, status: NXDOMAIN, ...

# Repeat across @dns.google and @dns.quad9.net for cross-resolver validation.
```

### Application-layer dangling-CDN proof (`cdn.openai.com` CRITICAL)

```bash
# DNS chain resolves cleanly:
dig +short @one.one.one.one CNAME cdn.openai.com
# → openaiassets.azureedge.net → openaiassets.afd.azureedge.net

# Provider returns deprovisioned-resource fingerprint at every path.
# Note: the AFD endpoint no longer presents a valid cert for its hostname,
# so curl needs -k to complete the TLS handshake. The cert error itself
# is corroborating evidence of deprovisioning.
curl -ksI https://openaiassets.afd.azureedge.net/
# HTTP/2 404
# x-azure-ref: 20260523T071201Z-...
# vary: Accept-Encoding

# Body fingerprint — Azure's generic "Page not found" HTML
# embedding the deprovisioned-resource reference URL:
curl -ks https://openaiassets.afd.azureedge.net/ | grep -i 'UE_404\|Page not found'
# → <title>Page not found</title>
# → <!-- saved from url=(0058)https://df.onecloud.azure-test.net/Error/UE_404?shown=true -->

# (Earlier package versions claimed the fingerprint was `<Code>ResourceNotFound</Code>`
#  XML — that's the Azure Storage / Blob fingerprint, not the AFD fingerprint.
#  Corrected 2026-05-23 after live re-verification.)

# Probe additional paths to rule out path-conditional 200s:
for p in / /test.png /v1/ /assets/ /static/; do
  printf "%s → " "$p"
  curl -kso /dev/null -w "%{http_code}\n" "https://openaiassets.afd.azureedge.net$p"
done
# All paths return 404.

# Cert-handshake check (additional deprovisioning signal):
echo | openssl s_client -connect openaiassets.afd.azureedge.net:443 \
  -servername openaiassets.afd.azureedge.net 2>/dev/null \
  | openssl x509 -noout -subject
# → subject does not match openaiassets.afd.azureedge.net (ERR_CERT_COMMON_NAME_INVALID)
```

### Registrar lookup (`rdap_lookup` tool)

Exposed at `https://dns-mcp.blackveilsecurity.com/mcp` — uses IANA RDAP bootstrap. Reproducible via raw RDAP:

```bash
# Direct IANA bootstrap-driven RDAP example (gTLDs):
curl -s "https://rdap.iana.org/domain/openai.io" | jq '.entities[] | select(.roles[]=="registrar") | .vcardArray[1][] | select(.[0]=="fn") | .[3]'
# → "GoDaddy.com, LLC"
```

### Email-auth posture per apex

```bash
APEX=openai.org
dig +short MX "$APEX"
dig +short TXT "$APEX" | grep -i 'v=spf1' || echo '(no SPF)'
dig +short TXT "_dmarc.$APEX" | grep -i 'v=DMARC1' || echo '(no DMARC)'
```

### Null-MX (RFC 7505) confirmation

```bash
APEX=anthropic.io
dig +short MX "$APEX"
# → 0 .
# (or "0 localhost." for the functionally-equivalent variant)
dig +short TXT "$APEX" | grep -E 'v=spf1.* -all'
# → "v=spf1 -all"
```

## Boundary conditions & honest caveats

A few things this package does **not** claim, recorded explicitly so they don't have to be teased out under questioning:

- **CloudFront / Azure-Container-Apps / AWS-NLB findings are operational drift, not active takeover vectors.** Targets embed provider-assigned random IDs (13–14-char distribution IDs, 8-hex-char env suffixes, 32-hex-char NLB prefixes) that are not practically reclaimable. They are real findings — the inventory is drifting — but the takeover-vector framing is reserved for the user-claimable AFD endpoint name `openaiassets`.
- **AWS has no published hostname-reservation policy for deleted NLBs.** The official NLB deletion docs (`docs.aws.amazon.com/elasticloadbalancing/latest/network/load-balancer-delete.html`, fetched 2026-05-23) document deletion mechanics but do **not** state a reservation window or a reuse-prevention guarantee for the deleted hostname. Practical takeover risk for the x.ai NLB cluster is therefore bounded by the pseudo-randomness of the 32-hex-char prefix + 10-digit suffix (~`16^32 · 10^10` search space) rather than by an AWS policy. We mark these MEDIUM, not CRITICAL, on that basis — honest about the absence of a positive guarantee on either side.
- **Azure's name-reservation policy applies to classic cloud services, not to Azure Front Door endpoints.** Microsoft's subdomain-takeover doc (`learn.microsoft.com/en-us/azure/security/fundamentals/subdomain-takeover`, fetched 2026-05-23) documents reservation only for `*.cloudapp.net` / `*.chinacloudapp.cn` / `*.usgovcloudapp.net` / `*.azurecloudapp.de` (classic cloud services). AFD endpoint names are explicitly not on that list, which is why the `openaiassets` reclassification as CRITICAL stands.
- **`c.ai` in the typosquat enumeration is Character.AI**, an unrelated established brand — surfaced by the edit-distance pass against `x.ai`, not a phishing-ready typosquat. Disclosed in § 6.
- **`openai.eu` declares null-MX but is parked at Sedo and is not OpenAI-owned.** Reproducibly resolves via Sedo's parking infrastructure; navigation lands on `sedo.com/search/details/?domain=openai.eu` with page title "openai.eu is available for purchase - Sedo.com" (screenshot in `.dev/demos/csc-evidence/screenshots/`). Not a posture finding for OpenAI.
- **Three brand-related apexes are registered-but-DNS-dark, not unregistered:** `openai.ai` (registrant `wangshaofei` in Shenzhen, registered 2017-12-16, status `inactive`), `openai.jp` (registered + mail-protected via `wats.gr.jp`), `anthropic.fr` (BLOOM'UP, no published DNS). Each was originally listed as an unregistered defensive gap; each was removed on direct registry RDAP.
- **`anthropic.dev` is blocked from open registration** by Google Registry's Brand Safety Alliance (BSA), not unregistered. Effectively unavailable to register today, but not a customer opportunity.
- **The earlier "CSC's portfolio inventory cannot see X" framing was a strawman.** CSC's actual product surface (per CSC public DomainSec and Subdomain Monitoring product pages, scraped 2026-05-23) includes Subdomain Monitoring on CSC-managed DNS, third-party-domain-registration trend monitoring, brand monitoring + phishing takedowns, and email-security tracking. The honest complementary axis is **vantage point** (CSC manages a defined surface; BlackVeil observes the public CT-log + RDAP surface), not capability presence/absence. The capability table in `01-exec-summary.md` was revised on 2026-05-23 to reflect this.
